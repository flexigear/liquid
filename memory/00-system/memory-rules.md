# Memory Rules

## Purpose

Keep project knowledge organized so later tasks can load the minimum useful context instead of re-reading everything.

## Principles

- Stable facts belong above volatile facts.
- Product truths should be separate from prototype details.
- A memory file should answer one class of question well.
- Retrieval should be task-driven, not chronology-driven.

## Layer Semantics

- `01-foundation`: Core identity, fantasy, constraints, and non-negotiables.
- `10-product`: Why the game should feel compelling and distinct.
- `11-research`: Comparable games and external references.
- `20-design`: Internal design judgments and gameplay conclusions.
- `30-prototype`: The current playable target and technical assumptions.
- `90-active`: What matters right now.

## When To Update

- Update `01-foundation` only when the game identity changes.
- Update `10-product` when the desired player experience changes.
- Update `20-design` when a design decision is validated or rejected.
- Update `30-prototype` when the target prototype scope or technical plan changes.
- Update `90-active` whenever the immediate work focus changes.

## Memory Hygiene

- Prefer editing an existing file over creating near-duplicate notes.
- Keep summaries short and specific.
- Record assumptions explicitly.
- Separate confirmed conclusions from open questions.
